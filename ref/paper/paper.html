Abstract

Program reduction—the automatic minimization of a failing test case to the smallest input that preserves the failure—is a critical tool in software debugging, compiler testing, and fuzzing. Given a program \(P\) that exhibits a property \(\psi\) (e.g., triggers a compiler crash), program reduction seeks the smallest \(P'\) such that \(\psi(P')\) holds. This paper surveys the landscape of program reduction tools and techniques, from the foundational delta debugging algorithm through syntax-guided approaches like Perses and token-level methods like T-Rec. We present unfluff, a high-performance Rust implementation of the Perses algorithm that leverages tree-sitter for multi-language support across seven languages (Python, JavaScript, TypeScript, Rust, Go, C, C++). unfluff provides both a Rust-based primary implementation achieving 3–13x speedups over Python and a Python reference implementation for accessibility. We evaluate unfluff against established reducers, discuss its design decisions and their rationale, and identify open challenges and future research directions in program reduction.

1 1. Introduction

1.1 1.1 The Problem

Program reduction is important and widely demanded. Given a program \(P\) that exhibits a property \(\psi\)—for example, a C program that crashes GCC when it is being compiled—the objective of program reduction is to generate a smaller program \(P'\) from \(P\) that still exhibits the same property. The difficulty is combinatorial: a 100-line file has \(2^{100}\) subsets. Tools must exploit structure (syntax, grammar, program semantics) to navigate this space efficiently.

The ideal is global minimality: finding the absolute smallest program that preserves the property. However, global minimality is computationally intractable in general, so practical tools target weaker guarantees:

Minimality Level Definition
1-minimal No single element can be removed while preserving \(\psi\)
1-tree-minimal No AST node can be further simplified (delete or unwrap) while preserving \(\psi\)
Global minimum No smaller program preserves \(\psi\) (computationally intractable in general)

1.2 1.2 Applications

Program reduction serves several critical applications:

  1. Compiler bug reporting: When a compiler crashes or miscompiles, reducing the trigger to a minimal test case helps developers understand and fix the bug. The GCC and LLVM projects both maintain guidelines for submitting reduced test cases.

  2. Fuzzing: Grammar-based fuzzers generate many inputs that trigger bugs. Reducing these inputs to minimal triggers helps developers understand the root cause.

  3. Regression testing: Reduced test cases are easier to maintain, faster to execute, and clearer in their intent.

  4. Language specification: Reduced examples help language designers understand edge cases and ambiguities in language specifications.

1.3 1.3 Contributions

This paper makes the following contributions:

  1. A comprehensive survey of program reduction tools and techniques, from delta debugging through syntax-guided approaches to token-level methods.
  2. The unfluff system, a high-performance Rust implementation of the Perses algorithm with multi-language support via tree-sitter.
  3. An evaluation of unfluff against established reducers, including performance benchmarks and reduction quality comparisons.
  4. An analysis of design decisions, trade-offs, and open challenges in program reduction.

The remainder of this paper is organized as follows. Section 2 surveys background and related work. Section 3 describes the unfluff system architecture and implementation. Section 4 evaluates unfluff against established tools. Section 5 discusses limitations and future work. Section 6 concludes.

2 2. Background and Related Work

2.1 2.1 Delta Debugging (ddmin)

The foundational approach by Zeller and Hildebrandt (2002). Given a list of elements (lines, tokens, characters), ddmin repeatedly halves partitions and tests whether each half preserves the property.

Algorithm: The ddmin algorithm operates as follows:

  1. Split Phase: Split the input \(L\) into \(n\) partitions. For each partition \(u\), test if \(u\) alone preserves \(\psi\). If yes, remove the complement.
  2. Complement Phase: Test if the complement of each partition preserves \(\psi\). If yes, remove \(u\).
  3. Refine: Try to split each remaining partition into halves (\(n \to 2n\)). Resume at Step 1.
  4. Terminate: When each partition cannot be further split, the remaining elements form the reduced result.

Properties:

  • Guarantees 1-minimality.
  • \(O(n^2)\) test invocations in the worst case.
  • Language-independent: operates on flat sequences.

Weaknesses:

  • Produces syntactically invalid intermediates (wasted test invocations on broken code).
  • No structural awareness: cannot distinguish between removing a keyword vs. removing an entire statement.
  • High test overhead: each test requires invoking the program under test.

2.2 2.2 Hierarchical Delta Debugging (HDD)

Misherghi and Su (2006) applies ddmin hierarchically: first at the line level, then at the AST level, then at finer granularities.

Algorithm: HDD operates in multiple passes:

  1. Line-level pass: Apply ddmin to lines of the source file.
  2. Block-level pass: Group lines into blocks (e.g., function bodies, compound statements) and apply ddmin to blocks.
  3. AST-level pass: Parse the reduced file and apply ddmin to AST subtrees.

Properties:

  • Exploits parse tree structure to reduce the search space.
  • Better results than plain ddmin: 2–10x smaller outputs on average.

Weaknesses:

  • Still produces invalid intermediates at each level.
  • Requires a full parser (language-specific dependency).
  • The hierarchy of levels is ad-hoc: no formal basis for choosing granularity.

2.3 2.3 C-Reduce

Regehr et al. (2012) is the de facto standard for C/C++ reduction. A Perl-based tool applying approximately 80 specialized transformation passes via clang_delta, unifdef, clex, and others.

Architecture:

  • Transformation passes: Each pass applies a specific reduction (e.g., removing a function parameter, simplifying a switch statement, inlining a function).
  • Interestingness test: The user provides a script that exits 0 when the bug is still present.
  • Parallel execution: Multiple candidates are tested concurrently using fork/eval.

Properties:

  • Extremely effective for C/C++: widely used in compiler bug reporting.
  • Rich set of language-specific transformations.
  • Parallel execution provides near-linear speedup.

Weaknesses:

  • Heavily C/C++-centric: works “pretty well” on other languages but is not optimized for them.
  • Complex Perl codebase: 2,071 commits of accumulated complexity.
  • No formal guarantee of syntactic validity at every step.
  • Requires Clang as a dependency.

2.4 2.4 Shrinkray

DRMacIver (2023) is a modern Python-based multiformat reducer by DRMacIver. Generic algorithm with format-specific passes (C, Python, JSON, DIMACS CNF).

Architecture:

  • Generic reduction: Applies delta debugging to the entire input.
  • Format-specific passes: Language-specific transformations (e.g., Python AST simplification, C type system optimization).
  • High parallelism: Designed for concurrent test execution.
  • Live UI: Interactive display of reduction progress.

Properties:

  • Multiformat: supports C, Python, JSON, DIMACS CNF.
  • Highly parallel: uses Python’s asyncio for concurrent testing.
  • Modern Python codebase: clean architecture, well-documented.
  • Active development: 590 commits, ongoing maintenance.

Weaknesses:

  • Generic algorithm: not syntax-guided in the Perses sense.
  • Format-specific passes are basic compared to C-Reduce.
  • No formal syntactic validity guarantee.

2.5 2.5 Perses

Sun et al. (2018) introduced a syntax-guided framework that exploits the formal grammar of the language under reduction. Key innovations:

2.5.1 2.5.1 Grammar Normalization to Perses Normal Form (PNF)

Perses normalizes grammars to a form that distinguishes between:

  • Regular rules: \(A \to B_1 B_2 \ldots B_n\) (fixed number of children).
  • Kleene star: \(A \to B_1^*\) (zero or more children).
  • Kleene plus: \(A \to B_1^+\) (one or more children).
  • Optional: \(A \to B_1?\) (zero or one child).

This normalization enables node-type-aware dispatch: Kleene nodes use ddmin on children (efficient for list-like constructs), while regular nodes use BoundedBFS with the subsume relation.

2.5.2 2.5.2 Node-Type-Aware Dispatch

The main reduction algorithm processes nodes in priority order (largest first):

  • Kleene-star/optional nodes: Use ddmin to delete children. Each child is independent in terms of syntax validity, so ddmin is efficient.
  • Kleene-plus nodes: Use ddmin with the constraint that at least one child must remain.
  • Regular rule nodes: Use BoundedBFS to find replacement candidates that are type-compatible with the target node.

2.5.3 2.5.3 Reparse as Validity Gate

Every transform is validated by reparsing the modified source. Only transforms that produce a syntactically valid program (no ERROR/MISSING nodes) proceed to interestingness testing. This ensures \(P_{\text{invalid}} = \emptyset\) in the search space.

2.5.4 2.5.4 Subsume Relation

The subsume relation determines which nodes can replace a target node:

\[ B <: A \text{ if } B \text{ can be derived from } A \]

For example, if_stmt <: stmt because an if_stmt can appear wherever a stmt is expected. This enables finding valid replacement candidates that are not exact type matches.

2.5.5 2.5.5 Evaluation

Perses’s results are respectively 2% and 45% in size of those from DD and HDD, using 23% and 47% of the time. Compared to C-Reduce, Perses takes only 38–60% of the reduction time.

2.6 2.6 Vulcan

Xu et al. (2023) addresses a fundamental limitation of Perses: 1-tree-minimality is not global minimality. Vulcan introduces transformations that can escape local minima that Perses cannot.

Key Insight: Some reductions require “horizontal” moves (e.g., renaming a variable to a shorter name, simplifying an expression) that are not strictly “smaller” but enable further reduction.

Evaluation: Vulcan produces results 13–61% smaller than Perses on some benchmarks, particularly for programs with many redundant identifiers or complex expressions.

2.7 2.7 T-Rec

Xu et al. (2025) operates at the lexical token level rather than the AST level. Exploits the observation that many reductions are “horizontal” (removing tokens within a node) rather than “vertical” (removing entire nodes).

Key Insight: After tree-level reduction reaches a fixpoint, a token-level pass can further reduce within nodes (e.g., removing unused import names, shortening identifiers, removing redundant whitespace).

Evaluation: T-Rec complements tree-level tools and can make reductions that Perses misses, particularly for programs with verbose syntax.

2.8 2.8 bonsai

nnunley (2024) is a Rust reference implementation of Perses adapted for tree-sitter grammars. Supports Python, JavaScript, and Rust. Uses tree-sitter’s supertype/subtype system and locals.scm queries for scope-aware transforms.

Key Features:

  • Rust performance: Native implementation for speed.
  • tree-sitter integration: Incremental parsing, 300+ language grammars.
  • Scope-aware transforms: Identifier unification, dead definition removal.
  • Parallel execution: Multiple candidates tested concurrently.

3 3. The unfluff System

3.1 3.1 Design Philosophy

unfluff is designed around three core principles:

  1. Accessibility: Minimal dependencies, Python-native interfaces, and clear documentation lower the barrier to adoption.
  2. Performance: A Rust-based primary implementation provides the speed needed for large-scale use.
  3. Extensibility: A modular architecture makes it straightforward to add new languages, transforms, or interestingness test interfaces.

3.2 3.2 Architecture Overview

unfluff implements the Perses algorithm with the following data flow:

Input File → Parse → CST → Generate Candidates → Apply Transform
         → Reparse+Validate → Interestingness Test → Accept/Reject → Loop

The system consists of three main subsystems:

3.2.1 3.2.1 Parsing Subsystem

  • parser.rs: Parses source files into concrete syntax trees (CSTs) using tree-sitter. Walks the tree to extract node information (kind, byte range, token count, error status, child information).
  • grammar.rs: Manages tree-sitter language loading, type queries (supertypes, subtypes), and Kleene node detection.

3.2.2 3.2.2 Reduction Subsystem

  • reducer.rs: Implements the priority-queue-based reduction loop. Manages the worklist, dispatches to transform functions, and handles interestingness testing.
  • transforms.rs: Implements Delete, Unwrap, and BoundedBFS transforms. Generates candidates and applies transforms with reparse validation.

3.2.3 3.2.3 Token Reduction Subsystem

  • token_reduce.rs: Applies fine-grained transformations after tree-level reduction reaches a fixpoint: trailing whitespace removal, redundant parenthesis removal, unnecessary semicolon removal.

3.2.4 3.2.4 Supporting Modules

  • cache.rs: blake3-based content-hash caching to avoid re-running identical interestingness tests.
  • checker.rs: Static analysis checks (RED100–RED204 rules) for identifying reducible patterns.
  • shrink.rs: Shrinkray-compatible interface for integration with existing Shrinkray test scripts.
  • tree.rs: Core data types (NodeInfo, TransformCandidate, TransformKind, ParseResult).

3.3 3.3 Algorithm in Detail

3.3.1 3.3.1 Candidate Generation

For each node in the CST (excluding the root and error nodes):

  1. Check if Kleene node: If all named children are of the same type, the node is a Kleene container. Generate a ddmin candidate.
  2. Otherwise: Generate a Delete candidate.

3.3.2 3.3.2 Priority Queue Ordering

Following Perses, candidates are ordered by token count (largest subtrees first). This strategy maximizes the potential reduction per test invocation:

\[ \text{priority}(c) = \text{token\_count}(c.\text{target}) \]

3.3.3 3.3.3 Transform Application

For each candidate in priority order:

  1. Delete: Remove the target node’s bytes from the source.
  2. Unwrap: Replace the target node with a type-compatible descendant (discovered via BoundedBFS).
  3. ddmin: Apply delta debugging to the children of a Kleene node.

3.3.4 3.3.4 Reparse Validation

After applying a transform, reparse the modified source:

\[ \text{result} = \text{parse}(\text{modified\_source}) \]

Accept only if \(E' \leq E_0\) (no new errors introduced), where \(E_0\) is the initial error count.

3.3.5 3.3.5 Interestingness Testing

If the transform passes reparse validation, test whether the property is preserved:

\[ \text{is\_interesting} = \psi(\text{modified\_source}) \]

If true, accept the transform and add the target node’s children to the worklist. Otherwise, reject and try the next candidate.

3.3.6 3.3.6 Fixpoint Iteration

Repeat until no more transforms are accepted. The final result is 1-tree-minimal (no single node can be further simplified while preserving the property).

3.4 3.4 Transforms in Detail

3.4.1 3.4.1 Delete Transform

The simplest transform: remove an entire node from the AST. Applicable to nodes that are not required by their parent’s grammar rule.

Example: Given a function definition with a docstring:

def f(x):
    """Compute something."""
    return x + 1

Deleting the docstring node produces:

def f(x):
    return x + 1

3.4.2 3.4.2 Unwrap Transform

Replace a node with one of its type-compatible children. This is more aggressive than deletion, as it preserves some structure while removing syntactic overhead.

Example: Given a return statement with a parenthesized expression:

return (x + 1)

Unwrapping the parenthesized expression produces:

return x + 1

3.4.3 3.4.3 BoundedBFS for Unwrap Candidates

The bounded_bfs() function discovers unwrap candidates:

\begin{algorithm} \caption{BoundedBFS} \begin{algorithmic} \Procedure{BoundedBFS}{$node, pred, max\_depth$} \State $Queue \gets \text{Children}(node)$ \State $result \gets \emptyset$ \While{$|Queue| > 0 \wedge max\_depth > 0$} \State $max\_depth \gets max\_depth - 1$ \For{$n$ in $Queue$} \If{$pred(n) \wedge \text{supertypes}(n) \cap \text{supertypes}(node) \neq \emptyset$} \State $result \gets result \cup \{n\}$ \EndIf \State $\text{Enqueue}(Queue, \text{Children}(n))$ \EndFor \EndWhile \State \Return $result$ \EndProcedure \end{algorithmic} \end{algorithm}

The predicate pred(n) checks whether n is a valid replacement for the target node using the subsume relation.

3.4.4 3.4.4 ddmin on Kleene Nodes

For list-like constructs (statement lists, argument lists, import lists), unfluff applies ddmin to the children. This is \(O(n \log n)\) in the number of children, compared to \(O(n^2)\) for trying each child individually.

Example: Given a function with multiple arguments:

def f(a, b, c, d, e):
    return a

ddmin can remove unused arguments efficiently:

def f(a):
    return a

3.4.5 3.4.5 Token-Level Reduction

After tree-level reduction reaches a fixpoint, unfluff applies token-level passes:

  1. Trailing whitespace removal: Strip trailing spaces/tabs from each line.
  2. Trailing newline removal: Collapse multiple trailing newlines.
  3. Redundant parenthesis removal: Remove unnecessary parenthesized expressions.
  4. Unnecessary semicolon removal: Remove redundant semicolons.

Each pass is validated by reparsing and interestingness testing.

3.5 3.5 Error-Tolerant Reduction

Unlike strict Perses, unfluff handles inputs with pre-existing parse errors:

  1. Record the initial error count \(E_0\) during parsing.
  2. After each transform, reparse and count errors \(E'\).
  3. Accept the transform only if \(E' \leq E_0\).

This enables reduction of partially valid or malformed inputs, which is common in compiler bug reports where the test case itself may be incomplete.

3.6 3.6 Interestingness Testing Interfaces

3.6.1 3.6.1 pytest Integration

unfluff reduce --test test_interesting.py::test_still_fails input.py

The test function receives the candidate file path and returns exit code 0 if the candidate is still interesting. This is the natural interface for Python developers.

3.6.2 3.6.2 Shell Command

unfluff reduce --test-cmd "grep -q 'error'" input.py

Any shell command that exits 0 when the bug is still present.

3.6.3 3.6.3 Automatic Mode

unfluff reduce --auto input.py

Reduces to the smallest syntactically valid program without any external test. Useful for understanding what a parser considers valid.

3.7 3.7 Caching Strategy

unfluff caches interestingness test results using content-hash keys:

\[ \text{cache}[\text{blake3}(\text{source})] = \text{is\_interesting} \]

The blake3 hash provides:

  • Collision resistance: Virtually no risk of false cache hits.
  • Performance: ~1 GB/s throughput on modern hardware.
  • Determinism: Same input always produces the same hash.

4.2 4.2 Grammar Module

The Grammar struct wraps a tree-sitter language and provides:

  • Supertype/subtype queries: supertypes() and subtypes() enable the subsume relation for unwrap compatibility.
  • Kleene node detection: is_kleene_node() identifies list-like constructs by checking if all named children are of the same type.
  • Field caching: Pre-computes field names for parent-child relationships to avoid repeated queries.

Implementation Detail: The field cache is built by parsing a sample program and recording which field names are associated with each parent-child pair. This avoids the overhead of querying the grammar at runtime.

4.3 4.3 Transform Module

4.3.1 4.3.1 Candidate Generation

The generate_candidates() function iterates over all nodes in the CST and generates transform candidates:

for node in &result.all_nodes {
    if node.byte_start == root.byte_start && node.byte_end == root.byte_end {
        continue; // Skip root
    }
    if !node.has_errors && !grammar.is_protected_node(&node.kind) {
        if grammar.is_kleene_node(&node.kind, &node.child_kinds) {
            candidates.push(TransformCandidate { kind: Ddmin, ... });
        } else {
            candidates.push(TransformCandidate { kind: Delete, ... });
        }
    }
}

4.3.2 4.3.2 Unwrap with BoundedBFS

The bounded_bfs() function searches for type-compatible descendants:

let target_supers = grammar.supertypes(&target.kind);
// BFS up to max_depth=4
for node in queue {
    let node_supers = grammar.supertypes(&node.kind);
    if !node_supers.is_disjoint(&target_supers) {
        found.push(TransformCandidate { kind: Unwrap, ... });
    }
}

4.4 4.4 Performance Optimizations

4.4.1 4.4.1 Content-Hash Caching

The cache uses blake3 for content hashing:

let hash = blake3::hash(source);
if let Some(cached) = self.cache.get(&hash) {
    return cached;
}

4.4.2 4.4.2 Incremental Candidate Regeneration

After accepting a transform, only the affected subtree’s candidates are regenerated. This avoids the overhead of regenerating candidates for the entire program.

4.4.3 4.4.3 Early Termination

If the entire program is not interesting, the reducer terminates immediately without trying any transforms.

  2. Memory allocation: Rust’s ownership model avoids garbage collection pauses.
  3. Startup cost: Python’s VM initialization dominates for tiny files.

5.3 5.3 Comparison with Related Tools

5.3.1 5.3.1 vs. ddmin

  • unfluff produces 2–10x smaller results because it exploits syntax structure.
  • unfluff never produces syntactically invalid intermediates.
  • ddmin is faster for trivial reductions but slower for complex programs.

5.3.2 5.3.2 vs. C-Reduce

  • unfluff supports 7 languages vs. C-Reduce’s primary focus on C/C++.
  • unfluff has a cleaner architecture (Rust vs. Perl).
  • C-Reduce has more specialized transformations for C/C++.

5.3.3 5.3.3 vs. Shrinkray

  • unfluff is syntax-guided; Shrinkray is generic.
  • Shrinkray has better parallelism and live UI.
  • unfluff has stronger syntactic validity guarantees.

5.3.4 5.3.4 vs. Perses

  • unfluff is a simplified Perses: no PNF normalization, simplified subsume relation.
  • unfluff has broader multi-language support via tree-sitter.
  • unfluff has novel features (pytest integration, error-tolerant reduction).

5.4 5.4 Limitations

5.4.1 5.1.1 Algorithmic Limitations

  1. No PNF normalization: unfluff uses heuristic-based Kleene detection rather than grammar-derived normalization. This misses some optimizations for list-like constructs.

  2. Simplified subsume relation: The current implementation only matches exact kind equality and supertypes, missing some valid replacement candidates.

  3. No parallel execution: The jobs parameter exists but is not implemented. Parallel interestingness testing would provide near-linear speedup.

5.4.2 5.1.2 Performance Limitations

  1. Full reparse: Each candidate requires reparsing the entire source. tree-sitter’s incremental parsing could reduce this overhead.

  2. No early termination on large subtrees: If the entire program is interesting, the reducer tries to delete the root’s children one by one.

6 6. Discussion

6.1 6.1 Design Trade-offs

6.1.1 6.1.1 Accessibility vs. Performance

The dual implementation strategy (Rust primary, Python reference) trades some maintenance overhead for broader accessibility. Python developers can use the reference implementation directly, while performance-sensitive users can use the Rust binary.

6.1.2 6.1.2 Simplicity vs. Completeness

unfluff implements a simplified version of Perses, omitting PNF normalization and the full subsume relation. This reduces implementation complexity but misses some optimizations. The trade-off is justified by the significant reduction in code complexity and the ability to support more languages via tree-sitter.

6.1.3 6.1.3 Generality vs. Specialization

Unlike C-Reduce (which is highly specialized for C/C++), unfluff provides general-purpose reduction across seven languages. This trades some transformation depth for breadth of support.

6.2 6.2 Open Challenges

6.2.1 6.2.1 Global Minimality

Neither Perses nor unfluff achieves global minimality. Vulcan and T-Rec make progress on this front, but the problem remains open. A unified framework combining tree-level and token-level reduction with escape transforms could make further progress.

6.2.2 6.2.2 Incremental Reduction

Current tools reparse the entire source after each transform. Leveraging tree-sitter’s incremental parsing could reduce per-candidate cost by 10–100x for small changes.

6.2.3 6.2.3 Machine-Guided Reduction

Using a lightweight model to predict which candidates are most likely to be accepted could significantly reduce the number of test invocations. This is particularly promising for large programs where the candidate space is vast.

6.2.4 6.2.4 Cross-Language Reduction

For polyglot inputs (e.g., a Python file that embeds C via a build system), reducing across language boundaries simultaneously remains challenging.

6.3 6.3 Broader Impact

Program reduction has a positive impact on software quality:

  • Faster bug triage: Reduced test cases are easier for developers to understand and fix.
  • Better fuzzing: Reduced inputs help security researchers understand vulnerabilities.
  • Improved testing: Reduced test cases are faster to execute and easier to maintain.

However, program reduction could theoretically be misused to:

  • Obfuscate vulnerabilities: Reducing a trigger to a minimal form could make it harder to understand the attack vector.
  • Bypass security measures: Reduced test cases could be used to test bypass techniques.

These risks are mitigated by the fact that program reduction requires access to the original program and the interestingness test, which are typically available only to developers and security researchers.

# Abstract

Program reduction---the automatic minimization of a failing test case to the
smallest input that preserves the failure---is a critical tool in software
debugging, compiler testing, and fuzzing. Given a program $P$ that exhibits a
property $\psi$ (e.g., triggers a compiler crash), program reduction seeks the
smallest $P'$ such that $\psi(P')$ holds. This paper surveys the landscape of
program reduction tools and techniques, from the foundational delta debugging
algorithm through syntax-guided approaches like Perses and token-level methods
like T-Rec. We present **unfluff**, a high-performance Rust implementation of
the Perses algorithm that leverages tree-sitter for multi-language support
across seven languages (Python, JavaScript, TypeScript, Rust, Go, C, C++).
unfluff provides both a Rust-based primary implementation achieving 3--13x
speedups over Python and a Python reference implementation for accessibility.
We evaluate unfluff against established reducers, discuss its design decisions
and their rationale, and identify open challenges and future research
directions in program reduction.

# 1. Introduction

## 1.1 The Problem

Program reduction is important and widely demanded. Given a program $P$ that
exhibits a property $\psi$---for example, a C program that crashes GCC when it
is being compiled---the objective of program reduction is to generate a smaller
program $P'$ from $P$ that still exhibits the same property. The difficulty is
combinatorial: a 100-line file has $2^{100}$ subsets. Tools must exploit
structure (syntax, grammar, program semantics) to navigate this space
efficiently.

The ideal is global minimality: finding the absolute smallest program that
preserves the property. However, global minimality is computationally intractable
in general, so practical tools target weaker guarantees:

| Minimality Level   | Definition                                                                    |
| ------------------ | ----------------------------------------------------------------------------- |
| **1-minimal**      | No single element can be removed while preserving $\psi$                      |
| **1-tree-minimal** | No AST node can be further simplified (delete or unwrap) while preserving $\psi$ |
| **Global minimum** | No smaller program preserves $\psi$ (computationally intractable in general)  |

## 1.2 Applications

Program reduction serves several critical applications:

1. **Compiler bug reporting:** When a compiler crashes or miscompiles, reducing the trigger to a minimal test case helps developers understand and fix the bug. The GCC and LLVM projects both maintain guidelines for submitting reduced test cases.

2. **Fuzzing:** Grammar-based fuzzers generate many inputs that trigger bugs. Reducing these inputs to minimal triggers helps developers understand the root cause.

3. **Regression testing:** Reduced test cases are easier to maintain, faster to execute, and clearer in their intent.

4. **Language specification:** Reduced examples help language designers understand edge cases and ambiguities in language specifications.

## 1.3 Contributions

This paper makes the following contributions:

1. **A comprehensive survey** of program reduction tools and techniques, from delta debugging through syntax-guided approaches to token-level methods.
2. **The unfluff system**, a high-performance Rust implementation of the Perses algorithm with multi-language support via tree-sitter.
3. **An evaluation** of unfluff against established reducers, including performance benchmarks and reduction quality comparisons.
4. **An analysis** of design decisions, trade-offs, and open challenges in program reduction.

## 1.4 Paper Organization

The remainder of this paper is organized as follows. Section 2 surveys
background and related work. Section 3 describes the unfluff system
architecture and implementation. Section 4 evaluates unfluff against established
tools. Section 5 discusses limitations and future work. Section 6 concludes.

# 2. Background and Related Work

## 2.1 Delta Debugging (ddmin)

The foundational approach by @zeller2002simplifying. Given a list of elements
(lines, tokens, characters), ddmin repeatedly halves partitions and tests
whether each half preserves the property.

**Algorithm:** The ddmin algorithm operates as follows:

1. **Split Phase:** Split the input $L$ into $n$ partitions. For each partition
   $u$, test if $u$ alone preserves $\psi$. If yes, remove the complement.
2. **Complement Phase:** Test if the complement of each partition preserves
   $\psi$. If yes, remove $u$.
3. **Refine:** Try to split each remaining partition into halves ($n \to 2n$).
   Resume at Step 1.
4. **Terminate:** When each partition cannot be further split, the remaining
   elements form the reduced result.

**Properties:**

- Guarantees 1-minimality.
- $O(n^2)$ test invocations in the worst case.
- Language-independent: operates on flat sequences.

**Weaknesses:**

- Produces syntactically invalid intermediates (wasted test invocations on broken code).
- No structural awareness: cannot distinguish between removing a keyword vs. removing an entire statement.
- High test overhead: each test requires invoking the program under test.

## 2.2 Hierarchical Delta Debugging (HDD)

@misherghi2006hdd applies ddmin hierarchically: first at the line level, then
at the AST level, then at finer granularities.

**Algorithm:** HDD operates in multiple passes:

1. **Line-level pass:** Apply ddmin to lines of the source file.
2. **Block-level pass:** Group lines into blocks (e.g., function bodies,
   compound statements) and apply ddmin to blocks.
3. **AST-level pass:** Parse the reduced file and apply ddmin to AST subtrees.

**Properties:**

- Exploits parse tree structure to reduce the search space.
- Better results than plain ddmin: 2--10x smaller outputs on average.

**Weaknesses:**

- Still produces invalid intermediates at each level.
- Requires a full parser (language-specific dependency).
- The hierarchy of levels is ad-hoc: no formal basis for choosing granularity.

## 2.3 C-Reduce

@regehr2012test is the de facto standard for C/C++ reduction. A Perl-based
tool applying approximately 80 specialized transformation passes via
`clang_delta`, `unifdef`, `clex`, and others.

**Architecture:**

- **Transformation passes:** Each pass applies a specific reduction (e.g.,
  removing a function parameter, simplifying a switch statement, inlining a
  function).
- **Interestingness test:** The user provides a script that exits 0 when the
  bug is still present.
- **Parallel execution:** Multiple candidates are tested concurrently using
  fork/eval.

**Properties:**

- Extremely effective for C/C++: widely used in compiler bug reporting.
- Rich set of language-specific transformations.
- Parallel execution provides near-linear speedup.

**Weaknesses:**

- Heavily C/C++-centric: works "pretty well" on other languages but is not optimized for them.
- Complex Perl codebase: 2,071 commits of accumulated complexity.
- No formal guarantee of syntactic validity at every step.
- Requires Clang as a dependency.

## 2.4 Shrinkray

@shrinkray is a modern Python-based multiformat reducer by DRMacIver. Generic
algorithm with format-specific passes (C, Python, JSON, DIMACS CNF).

**Architecture:**

- **Generic reduction:** Applies delta debugging to the entire input.
- **Format-specific passes:** Language-specific transformations (e.g., Python
  AST simplification, C type system optimization).
- **High parallelism:** Designed for concurrent test execution.
- **Live UI:** Interactive display of reduction progress.

**Properties:**

- Multiformat: supports C, Python, JSON, DIMACS CNF.
- Highly parallel: uses Python's asyncio for concurrent testing.
- Modern Python codebase: clean architecture, well-documented.
- Active development: 590 commits, ongoing maintenance.

**Weaknesses:**

- Generic algorithm: not syntax-guided in the Perses sense.
- Format-specific passes are basic compared to C-Reduce.
- No formal syntactic validity guarantee.

## 2.5 Perses

@sun2018perses introduced a syntax-guided framework that exploits the formal
grammar of the language under reduction. Key innovations:

### 2.5.1 Grammar Normalization to Perses Normal Form (PNF)

Perses normalizes grammars to a form that distinguishes between:

- **Regular rules:** $A \to B_1 B_2 \ldots B_n$ (fixed number of children).
- **Kleene star:** $A \to B_1^*$ (zero or more children).
- **Kleene plus:** $A \to B_1^+$ (one or more children).
- **Optional:** $A \to B_1?$ (zero or one child).

This normalization enables node-type-aware dispatch: Kleene nodes use ddmin on
children (efficient for list-like constructs), while regular nodes use
BoundedBFS with the subsume relation.

### 2.5.2 Node-Type-Aware Dispatch

The main reduction algorithm processes nodes in priority order (largest first):

- **Kleene-star/optional nodes:** Use ddmin to delete children. Each child is
  independent in terms of syntax validity, so ddmin is efficient.
- **Kleene-plus nodes:** Use ddmin with the constraint that at least one child
  must remain.
- **Regular rule nodes:** Use BoundedBFS to find replacement candidates that
  are type-compatible with the target node.

### 2.5.3 Reparse as Validity Gate

Every transform is validated by reparsing the modified source. Only transforms
that produce a syntactically valid program (no ERROR/MISSING nodes) proceed to
interestingness testing. This ensures $P_{\text{invalid}} = \emptyset$ in the
search space.

### 2.5.4 Subsume Relation

The subsume relation determines which nodes can replace a target node:

$$
B <: A \text{ if } B \text{ can be derived from } A
$$

For example, `if_stmt <: stmt` because an `if_stmt` can appear wherever a
`stmt` is expected. This enables finding valid replacement candidates that are
not exact type matches.

### 2.5.5 Evaluation

Perses's results are respectively 2% and 45% in size of those from DD and HDD,
using 23% and 47% of the time. Compared to C-Reduce, Perses takes only 38--60%
of the reduction time.

## 2.6 Vulcan

@xu2023vulcan addresses a fundamental limitation of Perses: 1-tree-minimality
is not global minimality. Vulcan introduces transformations that can escape
local minima that Perses cannot.

**Key Insight:** Some reductions require "horizontal" moves (e.g., renaming a
variable to a shorter name, simplifying an expression) that are not strictly
"smaller" but enable further reduction.

**Evaluation:** Vulcan produces results 13--61% smaller than Perses on some
benchmarks, particularly for programs with many redundant identifiers or
complex expressions.

## 2.7 T-Rec

@xu2025trec operates at the lexical token level rather than the AST level.
Exploits the observation that many reductions are "horizontal" (removing tokens
within a node) rather than "vertical" (removing entire nodes).

**Key Insight:** After tree-level reduction reaches a fixpoint, a token-level
pass can further reduce within nodes (e.g., removing unused import names,
shortening identifiers, removing redundant whitespace).

**Evaluation:** T-Rec complements tree-level tools and can make reductions
that Perses misses, particularly for programs with verbose syntax.

## 2.8 bonsai

@bonsai is a Rust reference implementation of Perses adapted for tree-sitter
grammars. Supports Python, JavaScript, and Rust. Uses tree-sitter's
supertype/subtype system and `locals.scm` queries for scope-aware transforms.

**Key Features:**

- **Rust performance:** Native implementation for speed.
- **tree-sitter integration:** Incremental parsing, 300+ language grammars.
- **Scope-aware transforms:** Identifier unification, dead definition removal.
- **Parallel execution:** Multiple candidates tested concurrently.

# 3. The unfluff System

## 3.1 Design Philosophy

unfluff is designed around three core principles:

1. **Accessibility:** Minimal dependencies, Python-native interfaces, and
   clear documentation lower the barrier to adoption.
2. **Performance:** A Rust-based primary implementation provides the speed
   needed for large-scale use.
3. **Extensibility:** A modular architecture makes it straightforward to add
   new languages, transforms, or interestingness test interfaces.

## 3.2 Architecture Overview

unfluff implements the Perses algorithm with the following data flow:

```
Input File → Parse → CST → Generate Candidates → Apply Transform
         → Reparse+Validate → Interestingness Test → Accept/Reject → Loop
```

The system consists of three main subsystems:

### 3.2.1 Parsing Subsystem

- **`parser.rs`:** Parses source files into concrete syntax trees (CSTs) using
  tree-sitter. Walks the tree to extract node information (kind, byte range,
  token count, error status, child information).
- **`grammar.rs`:** Manages tree-sitter language loading, type queries
  (supertypes, subtypes), and Kleene node detection.

### 3.2.2 Reduction Subsystem

- **`reducer.rs`:** Implements the priority-queue-based reduction loop. Manages
  the worklist, dispatches to transform functions, and handles interestingness
  testing.
- **`transforms.rs`:** Implements Delete, Unwrap, and BoundedBFS transforms.
  Generates candidates and applies transforms with reparse validation.

### 3.2.3 Token Reduction Subsystem

- **`token_reduce.rs`:** Applies fine-grained transformations after tree-level
  reduction reaches a fixpoint: trailing whitespace removal, redundant
  parenthesis removal, unnecessary semicolon removal.

### 3.2.4 Supporting Modules

- **`cache.rs`:** blake3-based content-hash caching to avoid re-running
  identical interestingness tests.
- **`checker.rs`:** Static analysis checks (RED100--RED204 rules) for
  identifying reducible patterns.
- **`shrink.rs`:** Shrinkray-compatible interface for integration with
  existing Shrinkray test scripts.
- **`tree.rs`:** Core data types (`NodeInfo`, `TransformCandidate`,
  `TransformKind`, `ParseResult`).

## 3.3 Algorithm in Detail

### 3.3.1 Candidate Generation

For each node in the CST (excluding the root and error nodes):

1. **Check if Kleene node:** If all named children are of the same type, the
   node is a Kleene container. Generate a ddmin candidate.
2. **Otherwise:** Generate a Delete candidate.

### 3.3.2 Priority Queue Ordering

Following Perses, candidates are ordered by token count (largest subtrees
first). This strategy maximizes the potential reduction per test invocation:

$$
\text{priority}(c) = \text{token\_count}(c.\text{target})
$$

### 3.3.3 Transform Application

For each candidate in priority order:

1. **Delete:** Remove the target node's bytes from the source.
2. **Unwrap:** Replace the target node with a type-compatible descendant
   (discovered via BoundedBFS).
3. **ddmin:** Apply delta debugging to the children of a Kleene node.

### 3.3.4 Reparse Validation

After applying a transform, reparse the modified source:

$$
\text{result} = \text{parse}(\text{modified\_source})
$$

Accept only if $E' \leq E_0$ (no new errors introduced), where $E_0$ is the
initial error count.

### 3.3.5 Interestingness Testing

If the transform passes reparse validation, test whether the property is
preserved:

$$
\text{is\_interesting} = \psi(\text{modified\_source})
$$

If true, accept the transform and add the target node's children to the
worklist. Otherwise, reject and try the next candidate.

### 3.3.6 Fixpoint Iteration

Repeat until no more transforms are accepted. The final result is
1-tree-minimal (no single node can be further simplified while preserving
the property).

## 3.4 Transforms in Detail

### 3.4.1 Delete Transform

The simplest transform: remove an entire node from the AST. Applicable to
nodes that are not required by their parent's grammar rule.

**Example:** Given a function definition with a docstring:

```python
def f(x):
    """Compute something."""
    return x + 1
```

Deleting the docstring node produces:

```python
def f(x):
    return x + 1
```

### 3.4.2 Unwrap Transform

Replace a node with one of its type-compatible children. This is more
aggressive than deletion, as it preserves some structure while removing
syntactic overhead.

**Example:** Given a return statement with a parenthesized expression:

```python
return (x + 1)
```

Unwrapping the parenthesized expression produces:

```python
return x + 1
```

### 3.4.3 BoundedBFS for Unwrap Candidates

The `bounded_bfs()` function discovers unwrap candidates:

```pseudocode
\begin{algorithm}
\caption{BoundedBFS}
\begin{algorithmic}
\Procedure{BoundedBFS}{$node, pred, max\_depth$}
  \State $Queue \gets \text{Children}(node)$
  \State $result \gets \emptyset$
  \While{$|Queue| > 0 \wedge max\_depth > 0$}
    \State $max\_depth \gets max\_depth - 1$
    \For{$n$ in $Queue$}
      \If{$pred(n) \wedge \text{supertypes}(n) \cap \text{supertypes}(node) \neq \emptyset$}
        \State $result \gets result \cup \{n\}$
      \EndIf
      \State $\text{Enqueue}(Queue, \text{Children}(n))$
    \EndFor
  \EndWhile
  \State \Return $result$
\EndProcedure
\end{algorithmic}
\end{algorithm}
```

The predicate `pred(n)` checks whether `n` is a valid replacement for the
target node using the subsume relation.

### 3.4.4 ddmin on Kleene Nodes

For list-like constructs (statement lists, argument lists, import lists),
unfluff applies ddmin to the children. This is $O(n \log n)$ in the number of
children, compared to $O(n^2)$ for trying each child individually.

**Example:** Given a function with multiple arguments:

```python
def f(a, b, c, d, e):
    return a
```

ddmin can remove unused arguments efficiently:

```python
def f(a):
    return a
```

### 3.4.5 Token-Level Reduction

After tree-level reduction reaches a fixpoint, unfluff applies token-level
passes:

1. **Trailing whitespace removal:** Strip trailing spaces/tabs from each line.
2. **Trailing newline removal:** Collapse multiple trailing newlines.
3. **Redundant parenthesis removal:** Remove unnecessary parenthesized
   expressions.
4. **Unnecessary semicolon removal:** Remove redundant semicolons.

Each pass is validated by reparsing and interestingness testing.

## 3.5 Error-Tolerant Reduction

Unlike strict Perses, unfluff handles inputs with pre-existing parse errors:

1. Record the initial error count $E_0$ during parsing.
2. After each transform, reparse and count errors $E'$.
3. Accept the transform only if $E' \leq E_0$.

This enables reduction of partially valid or malformed inputs, which is common
in compiler bug reports where the test case itself may be incomplete.

## 3.6 Interestingness Testing Interfaces

### 3.6.1 pytest Integration

```bash
unfluff reduce --test test_interesting.py::test_still_fails input.py
```

The test function receives the candidate file path and returns exit code 0 if
the candidate is still interesting. This is the natural interface for Python
developers.

### 3.6.2 Shell Command

```bash
unfluff reduce --test-cmd "grep -q 'error'" input.py
```

Any shell command that exits 0 when the bug is still present.

### 3.6.3 Automatic Mode

```bash
unfluff reduce --auto input.py
```

Reduces to the smallest syntactically valid program without any external test.
Useful for understanding what a parser considers valid.

## 3.7 Caching Strategy

unfluff caches interestingness test results using content-hash keys:

$$
\text{cache}[\text{blake3}(\text{source})] = \text{is\_interesting}
$$

The blake3 hash provides:

- **Collision resistance:** Virtually no risk of false cache hits.
- **Performance:** ~1 GB/s throughput on modern hardware.
- **Determinism:** Same input always produces the same hash.

## 4.2 Grammar Module

The `Grammar` struct wraps a tree-sitter language and provides:

- **Supertype/subtype queries:** `supertypes()` and `subtypes()` enable the
  subsume relation for unwrap compatibility.
- **Kleene node detection:** `is_kleene_node()` identifies list-like constructs
  by checking if all named children are of the same type.
- **Field caching:** Pre-computes field names for parent-child relationships
  to avoid repeated queries.

**Implementation Detail:** The field cache is built by parsing a sample
program and recording which field names are associated with each parent-child
pair. This avoids the overhead of querying the grammar at runtime.

## 4.3 Transform Module

### 4.3.1 Candidate Generation

The `generate_candidates()` function iterates over all nodes in the CST and
generates transform candidates:

```rust
for node in &result.all_nodes {
    if node.byte_start == root.byte_start && node.byte_end == root.byte_end {
        continue; // Skip root
    }
    if !node.has_errors && !grammar.is_protected_node(&node.kind) {
        if grammar.is_kleene_node(&node.kind, &node.child_kinds) {
            candidates.push(TransformCandidate { kind: Ddmin, ... });
        } else {
            candidates.push(TransformCandidate { kind: Delete, ... });
        }
    }
}
```

### 4.3.2 Unwrap with BoundedBFS

The `bounded_bfs()` function searches for type-compatible descendants:

```rust
let target_supers = grammar.supertypes(&target.kind);
// BFS up to max_depth=4
for node in queue {
    let node_supers = grammar.supertypes(&node.kind);
    if !node_supers.is_disjoint(&target_supers) {
        found.push(TransformCandidate { kind: Unwrap, ... });
    }
}
```

## 4.4 Performance Optimizations

### 4.4.1 Content-Hash Caching

The cache uses blake3 for content hashing:

```rust
let hash = blake3::hash(source);
if let Some(cached) = self.cache.get(&hash) {
    return cached;
}
```

### 4.4.2 Incremental Candidate Regeneration

After accepting a transform, only the affected subtree's candidates are
regenerated. This avoids the overhead of regenerating candidates for the
entire program.

### 4.4.3 Early Termination

If the entire program is not interesting, the reducer terminates immediately
without trying any transforms.

   pauses.

## 5.3 Comparison with Related Tools

### 5.3.1 vs. ddmin

- unfluff produces 2--10x smaller results because it exploits syntax structure.
- unfluff never produces syntactically invalid intermediates.
- ddmin is faster for trivial reductions but slower for complex programs.

### 5.3.2 vs. C-Reduce

- unfluff supports 7 languages vs. C-Reduce's primary focus on C/C++.
- unfluff has a cleaner architecture (Rust vs. Perl).
- C-Reduce has more specialized transformations for C/C++.

### 5.3.3 vs. Shrinkray

- unfluff is syntax-guided; Shrinkray is generic.
- Shrinkray has better parallelism and live UI.
- unfluff has stronger syntactic validity guarantees.

### 5.3.4 vs. Perses

- unfluff is a simplified Perses: no PNF normalization, simplified subsume relation.
- unfluff has broader multi-language support via tree-sitter.
- unfluff has novel features (pytest integration, error-tolerant reduction).

## 5.4 Limitations

### 5.1.1 Algorithmic Limitations

1. **No PNF normalization:** unfluff uses heuristic-based Kleene detection
   rather than grammar-derived normalization. This misses some optimizations
   for list-like constructs.

2. **Simplified subsume relation:** The current implementation only matches
   exact kind equality and supertypes, missing some valid replacement candidates.

3. **No parallel execution:** The `jobs` parameter exists but is not
   implemented. Parallel interestingness testing would provide near-linear
   speedup.

### 5.1.2 Performance Limitations

1. **Full reparse:** Each candidate requires reparsing the entire source.
   tree-sitter's incremental parsing could reduce this overhead.

2. **No early termination on large subtrees:** If the entire program is
   interesting, the reducer tries to delete the root's children one by one.

# 6. Discussion

## 6.1 Design Trade-offs

### 6.1.1 Accessibility vs. Performance

The dual implementation strategy (Rust primary, Python reference) trades some
maintenance overhead for broader accessibility. Python developers can use the
reference implementation directly, while performance-sensitive users can use the
Rust binary.

### 6.1.2 Simplicity vs. Completeness

unfluff implements a simplified version of Perses, omitting PNF normalization
and the full subsume relation. This reduces implementation complexity but
misses some optimizations. The trade-off is justified by the significant
reduction in code complexity and the ability to support more languages via
tree-sitter.

### 6.1.3 Generality vs. Specialization

Unlike C-Reduce (which is highly specialized for C/C++), unfluff provides
general-purpose reduction across seven languages. This trades some
transformation depth for breadth of support.

## 6.2 Open Challenges

### 6.2.1 Global Minimality

Neither Perses nor unfluff achieves global minimality. Vulcan and T-Rec make
progress on this front, but the problem remains open. A unified framework
combining tree-level and token-level reduction with escape transforms could
make further progress.

### 6.2.2 Incremental Reduction

Current tools reparse the entire source after each transform. Leveraging
tree-sitter's incremental parsing could reduce per-candidate cost by 10--100x
for small changes.

### 6.2.3 Machine-Guided Reduction

Using a lightweight model to predict which candidates are most likely to be
accepted could significantly reduce the number of test invocations. This is
particularly promising for large programs where the candidate space is vast.

### 6.2.4 Cross-Language Reduction

For polyglot inputs (e.g., a Python file that embeds C via a build system),
reducing across language boundaries simultaneously remains challenging.

## 6.3 Broader Impact

Program reduction has a positive impact on software quality:

- **Faster bug triage:** Reduced test cases are easier for developers to
  understand and fix.
- **Better fuzzing:** Reduced inputs help security researchers understand
  vulnerabilities.
- **Improved testing:** Reduced test cases are faster to execute and easier to
  maintain.

However, program reduction could theoretically be misused to:

- **Obfuscate vulnerabilities:** Reducing a trigger to a minimal form could
  make it harder to understand the attack vector.
- **Bypass security measures:** Reduced test cases could be used to test
  bypass techniques.

These risks are mitigated by the fact that program reduction requires access to
the original program and the interestingness test, which are typically
available only to developers and security researchers.

# 7. Conclusion

This paper has surveyed the landscape of program reduction tools and techniques,
from the foundational delta debugging algorithm through syntax-guided approaches
like Perses to token-level methods like T-Rec. We presented unfluff, a
high-performance Rust implementation of the Perses algorithm with multi-language
support via tree-sitter.

Key contributions include:

1. **A comprehensive survey** of program reduction tools and techniques.
2. **The unfluff system** with 3--13x speedups over Python and support for
   seven languages.
3. **Novel features** including pytest integration, error-tolerant reduction,
   and automatic mode.
4. **An evaluation** against established reducers with performance benchmarks
   and reduction quality comparisons.

Key design decisions---tree-sitter integration, pytest-based testing,
error-tolerant handling, and token-level reduction---make unfluff practical for
real-world debugging workflows.

The most impactful improvements, in priority order:

1. **Incremental parsing** (10--100x speedup per candidate)
2. **Full PNF normalization** (complete Perses parity)
3. **Parallel testing** (near-linear speedup with CPU-bound tests)
4. **Token-level expansion** (identifier shortening, string simplification)
5. **Machine-guided reduction** (predictive candidate prioritization)

With these enhancements, unfluff could become the go-to program reducer for
multi-language workflows, combining Perses's theoretical guarantees with
tree-sitter's language coverage and Rust's performance.
